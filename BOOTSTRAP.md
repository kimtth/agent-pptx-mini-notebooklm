## Business / Consulting Presentation Frameworks

These are not code frameworks. They are content-structuring frameworks used in strategy consulting, executive communication, and board-style presentations. If the goal is a McKinsey-style deck, these matter as much as the underlying `.pptx` generation library.

| Framework | Typical use | Why it matters for automation |
|----------|-------------|-------------------------------|
| SCQA (Situation, Complication, Question, Answer) | Executive storyline, proposal opening, recommendation decks | Gives an LLM a reliable narrative spine for the first 3-5 slides. Useful for turning research notes into a sharp consulting-style opener. |
| Pyramid Principle | Top-down argument structure | Forces the main answer first, then groups supporting arguments underneath. Strong fit for slide generation because each branch can map to a section or slide cluster. |
| MECE + Issue Tree | Problem decomposition and workstream planning | Helps break a business problem into non-overlapping buckets. Useful when generating agenda, analysis sections, and structured appendices. |
| Action-title slides | One message per slide, written as a conclusion | Core consulting habit: the slide title is the takeaway, not the topic. This is one of the fastest ways to make an AI-generated deck feel more like McKinsey, Bain, or BCG. |
| Assertion-Evidence | Decision memos, insight-heavy analytics slides | Replaces vague headers with a claim supported by chart/table evidence. Useful for automated chart-caption pairing and KPI slides. |
| Executive Summary first | Board, C-level, and investment decks | Encourages the deck to start with recommendation, impact, risks, and next steps before detail. Important when generating business decks for decision-makers. |
| Main story + appendix split | Consulting reports and client presentations | Keeps the core storyline short while moving backup analyses into appendix slides. Useful for agent pipelines that generate both narrative and reference material. |

## McKinsey-Style Deck Pattern

For a business perspective, a practical McKinsey-style presentation usually follows this flow:

1. Executive summary: the answer first, with 3 key takeaways.
2. Situation: what is happening in the market, company, or project.
3. Complication: what changed, what is broken, or what decision is required.
4. Key question: the decision or problem to solve.
5. Answer / recommendation: the proposed direction.
6. Supporting evidence: market data, financial model, customer insights, benchmarks.
7. Options and trade-offs: what alternatives were considered and why they were rejected.
8. Implementation roadmap: milestones, owners, timing, and risks.
9. Appendix: backup analyses, assumptions, and detailed exhibits.

In practice, the visual style is usually:

- Short action-oriented slide titles.
- Clean grids and strong alignment.
- 1 key message per slide.
- Evidence-heavy exhibits rather than decorative visuals.
- Consistent use of color only to emphasize meaning.

## Lessons Learned

### Tailwind CSS v4: Unlayered CSS Overrides All Utilities

**Problem:** In Tailwind CSS v4, `@import "tailwindcss"` places utilities inside `@layer utilities`. Per the CSS cascade layers specification, **unlayered styles always beat layered styles**. Writing bare resets like:

```css
@import "tailwindcss";
* { margin: 0; padding: 0; }
```

causes `margin: 0` and `padding: 0` to silently override **every** Tailwind spacing utility (`px-4`, `py-3`, `p-3`, `gap-3`, `mb-2`, `space-y-4`, etc.) across the entire app. Only inline `style={{ padding: '...' }}` takes effect, making all class-based spacing appear broken — elements overlap, labels stick to borders, buttons clip edges.

**Fix:** Always wrap custom base/reset styles in `@layer base` so that Tailwind's `@layer utilities` can properly override them:

```css
@import "tailwindcss";

@layer base {
  * { box-sizing: border-box; margin: 0; padding: 0; }
}
```

**Rule:** Never write unlayered CSS rules in a Tailwind v4 project. All custom styles must go in `@layer base`, `@layer components`, or `@layer utilities`.
